  <action>
  **Extend `src/lib/server/db/schema.ts`** with three new tables and their relations:

  1. **`personas` table** — `id` (uuid, pk), `userId` (uuid, FK → users.id, cascade delete), `name` (varchar 255, not null), `description` (text, nullable), `platform` (varchar 50, nullable — for visual identity icon/color only, NOT a usage constraint per user decision), `isDefault` (boolean, default false, not null), `archivedAt` (timestamp with tz, nullable — soft delete per user decision), `activeVoiceVersionId` (uuid, nullable — points to current active voice profile version), `createdAt`, `updatedAt` (timestamps with tz, defaultNow, not null). Add index on `userId`. Import `boolean`, `jsonb`, `index` from drizzle pg-core as needed.

  2. **`writingSamples` table** — `id` (uuid, pk), `personaId` (uuid, FK → personas.id, cascade delete), `content` (text, not null), `platform` (varchar 50, nullable — twitter, linkedin, blog, etc.), `wordCount` (integer, not null), `createdAt` (timestamp with tz, defaultNow, not null). Add index on `personaId`.

  3. **`voiceProfileVersions` table** — `id` (uuid, pk), `personaId` (uuid, FK → personas.id, cascade delete), `version` (integer, not null), `extractedProfile` (jsonb, not null — the full VoiceProfile JSON from AI extraction), `manualEdits` (jsonb, nullable — user overrides stored separately from extraction), `sampleCount` (integer, not null), `samplePlatforms` (jsonb, not null — e.g. ["twitter", "blog"]), `totalWordCount` (integer, not null), `createdAt` (timestamp with tz, defaultNow, not null). Add index on `personaId`. Do NOT add a unique index on `(personaId, platform)` — user decided personas span platforms, not per-platform.

  4. **Relations** — Add `personasRelations` (user → one, writingSamples → many, voiceProfileVersions → many), `writingSamplesRelations` (persona → one), `voiceProfileVersionsRelations` (persona → one). Also add `personas: many(personas)` to the existing `usersRelations`.

  **IMPORTANT:** Do NOT add a `uniqueIndex` on persona+platform for voice profiles — the architecture.md schema had this but user decided voice profiles version independently (not per-platform). See Research Pitfall 1.

  **Create `src/lib/schemas/persona.ts`** — Zod schemas:
  - `createPersonaSchema`: `name` (string, trim, min 1, max 255), `description` (string, max 2000, optional), `platform` (enum of "twitter" | "linkedin" | "blog" | "reddit" | "email" | "other", optional)
  - `editPersonaSchema`: same fields as create, all optional except at least one must be present

  **Create `src/lib/schemas/voice-profile.ts`** — Zod schema for the AI-extracted voice profile structure (used in Plan 02 for extraction AND for type safety when reading from DB):
  - `summary` (string), `tone` object (`primary` string, `secondary` string optional, `formality` enum), `vocabulary` object (`level` enum, `favorites` string array, `avoids` string array, `jargon` string array), `sentenceStructure` object (`averageLength` enum, `complexity` enum, `patterns` string array), `quirks` (string array), `rhetoric` object (`persuasionStyle` string, `transitions` string array, `openingStyle` string, `closingStyle` string)
  - Export the Zod schema AND the inferred TypeScript type

  After creating schema, run `bunx drizzle-kit push` to apply to local DB.

  **Create `src/lib/server/services/persona.ts`** — service functions that accept a Drizzle DB instance (from `getDb()`) and return persona operations. NOT a class, use a factory function pattern:

  ```typescript
  export function createPersonaService(db: Awaited<ReturnType<typeof getDb>>) { ... }
  ```

  Methods:
  - `list(userId)` — SELECT all non-archived personas for user, ordered by isDefault DESC then updatedAt DESC. Include a left join or subquery to get the active voice profile version's `extractedProfile.summary` for display.
  - `getById(userId, id)` — SELECT single persona with ownership check (userId match) including active voice profile version data.
  - `create(userId, data)` — INSERT persona with name (required), description and platform (optional). If this is the user's first persona, automatically set `isDefault: true`.
  - `update(userId, id, data)` — UPDATE name/description/platform with ownership check. Set `updatedAt` to now.
  - `archive(userId, id)` — SET `archivedAt` to now (soft delete per user decision). If archiving the default persona, set another persona as default (most recently updated).
  - `setDefault(userId, id)` — Unset all other defaults for user, then set this one. Two queries (no transaction needed for this simple operation on Workers).

  **Create `/personas` list page** (`+page.server.ts` + `+page.svelte`):
  - Server load: Get session userId, get DB via `getDb()` (same pattern as auth: `event.platform?.env?.DATABASE_URL ?? env.DATABASE_URL`), call `persona.list(userId)`, return personas array.
  - UI: Page title "Personas", "New persona" button (links to `/personas/new`). Grid of persona cards — each card shows: persona name (bold), description (truncated to 2 lines, gray text), voice summary if voice profile exists (small italic text like "Casual, witty, uses analogies") or "No voice profile" badge with CTA, platform icon/color indicator (colored dot or small icon — use Tailwind colors: twitter=sky-400, linkedin=blue-600, blog=emerald-500, reddit=orange-500, email=gray-500, other=gray-400), default badge (small "Default" pill if isDefault). Cards link to `/personas/[id]`. If no personas exist, show empty state with "Create your first persona" CTA.
  - Each card has a subtle "Default" badge if isDefault, and a "No voice profile — Add samples" nudge if no active voice version (per user decision: "UI nudges user to complete setup").

  **Create `/personas/new` quick-create page** (`+page.server.ts` + `+page.svelte`):
  - Form with: name input (required, placeholder "e.g., Professional Twitter"), description textarea (optional, placeholder "Brief description of this voice identity"), platform dropdown (optional: Twitter, LinkedIn, Blog, Reddit, Email, Other — per user decision for visual identity).
  - Form action: validate with `createPersonaSchema`, call `persona.create()`, redirect to `/personas/[id]`.
  - Quick create means name is the only required field — low friction per user decision.

  **Create `/personas/[id]` detail/edit page** (`+page.server.ts` + `+page.svelte`):
  - Server load: Get persona by id with ownership check. If not found or wrong user, redirect to `/personas`. Load persona + active voice profile version data.
  - UI layout: Top section — editable name and description (inline form with save button), platform selector. Below — voice profile section (placeholder for now, will be built in Plan 02; show "No voice profile yet — add writing samples to extract your voice" CTA that Plan 02 will replace with the extraction UI). Bottom — actions: "Set as default" button (if not already default), "Archive persona" button (red, with confirmation).
  - Form actions: `update` (name/description/platform), `archive` (soft delete, redirect to `/personas`), `setDefault` (set as default, stay on page).
  - All forms use SvelteKit `use:enhance` for progressive enhancement (consistent with Phase 1 auth patterns).

  **Update `src/routes/(app)/+layout.svelte`** — Add "Personas" link to the nav bar, next to the user email. Style as a nav link consistent with existing nav design.
