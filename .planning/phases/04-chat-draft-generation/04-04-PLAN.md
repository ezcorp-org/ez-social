**Create `src/routes/(app)/queue/[id]/drafts/+server.ts`** — API endpoint for saving draft edits:

1. Auth check: `locals.auth()` — 401 if no session
2. Parse JSON body: `{ messageId, originalText, editedText }`
3. Validate: all three fields required, editedText must differ from originalText
4. Per-request DB setup (same pattern as other +server.ts files)
5. Call `draftService.saveEdit({ messageId, postId: params.id, originalText, editedText })`
6. Return JSON `{ success: true, edit }` with 201 status

**Update `src/lib/components/chat/DraftBlock.svelte`** — Add inline editing:

The DraftBlock already has the draft text, copy button, and character/word count from Plan 03. Now add editing:

1. Add `postId` prop (string) — needed for the API call
2. Add state: `isEditing` ($state, false), `editText` ($state, ''), `isSaving` ($state, false), `hasBeenEdited` ($state, false)
3. Also accept a `savedEditText` prop (string | null) — if a prior edit exists (from draftEdits loaded on page), display the edited version instead of original, and show a small "Edited" badge
4. Edit button: When clicked, set `isEditing = true`, `editText = text` (copy original to edit buffer)
5. Editing UI: Replace the text display with a `<textarea>` (not contenteditable — more reliable) pre-filled with `editText`. Show Save and Cancel buttons below.
6. Cancel: Reset `isEditing = false`, discard editText
7. Save: POST to `/queue/${postId}/drafts` with `{ messageId, originalText: text, editedText: editText }`. On success, set `hasBeenEdited = true`, `isEditing = false`. Update displayed text to show edited version. On error, show inline error message.
8. When `hasBeenEdited` or `savedEditText` is truthy, show a small "Edited" label and display the edited text as the primary display (original still accessible via tooltip or small "Show original" toggle)
9. Update character/word counts to reflect the currently displayed text (edited if available, original otherwise)
  </action>

Update ChatInterface to handle persona switching as a full-featured interaction:

1. **handlePersonaSwitch function**: When PersonaSelector's `onSwitch` is called with a new personaId:
   a. Update `currentPersonaId` state — this automatically updates the Chat transport's dynamic body function
   b. Add a local system-style message to the chat display. Since the Chat class manages messages, we need to handle this carefully:
      - Create a visual-only system message in the UI: insert a divider/notification in the message list that says "Switched to [persona name] voice" — styled as a centered, muted message (similar to "system" role styling in ChatMessage)
      - To make this persist: POST to the chat endpoint with a user message like "Switch to [persona name] voice" — the server will detect the persona change from the body.personaId, update the post's persona assignment, and the AI can acknowledge the switch
      - Alternative simpler approach: Save a system-type message directly to the DB via a small helper fetch, then add it to the local messages array. BUT the Chat class owns messages — so the simplest approach is to just update `currentPersonaId` and let the next user message go through with the new persona. Add an inline notification between messages (not a real chat message, just a visual divider) using a $state array of persona-switch events keyed by timestamp.
   c. The SIMPLEST correct approach: Update `currentPersonaId`. Show a transient "Switched to [name]" notification near the persona selector (fade out after 3 seconds). The next message the user sends will use the new persona's context. The server already handles personaId in the request body and will update the post's persona assignment + rebuild the system prompt with the new voice.

2. **Persona switch notification**: Add a small notification area near the PersonaSelector. When persona switches, show "Now replying as [name]" with a brief fade-in/out animation (use $state + setTimeout).

3. **Pass postId to DraftBlock**: Ensure ChatMessage passes `postId` and the relevant `draftEdits` for each message to DraftBlock components. The `draftEdits` array from page data needs to be matched by messageId to the corresponding DraftBlock.

4. **Map draftEdits to messages**: In ChatInterface, create a $derived map: `editsByMessageId` that groups draftEdits by messageId. Pass the relevant edits when rendering ChatMessage components so DraftBlock can show previously saved edits.
