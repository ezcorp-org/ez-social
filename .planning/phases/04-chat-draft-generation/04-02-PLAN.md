  <name>Task 1: Create system prompt builder</name>
  <files>src/lib/server/chat-prompt.ts</files>
  <action>
Create a system prompt builder function that assembles the full AI context for the chat. This is a pure function, no DB access.

```typescript
export function buildChatSystemPrompt(context: {
  post: { url: string; platform: string | null; postContent: string | null; postAuthor: string | null };
  persona: { name: string } | null;
  voiceProfile: { extractedProfile: unknown; manualEdits: unknown } | null;
}): string
```

The system prompt must:

1. **Role definition**: Instruct the AI that it's a social media reply assistant helping the user craft replies in their persona's voice.

2. **Original post context**: Include the full post content, author, platform, and URL so the AI can discuss it intelligently.

3. **Voice profile**: If a voice profile exists, include the merged profile (extractedProfile with manualEdits overlaid). Instruct the AI to match these voice characteristics when generating drafts but to converse naturally when discussing the post.

4. **Draft format instructions**: Tell the AI to wrap draft reply text in `<draft>...</draft>` markers when the user requests a draft. The draft should contain ONLY the reply text (what would be posted), not meta-commentary. Outside of draft markers, the AI can explain, discuss, or provide context.

5. **Behavioral instructions**:
   - Be responsive to both discussion requests and draft requests — no rigid flow
   - When asked to refine a draft ("make it shorter", "more casual"), generate a new `<draft>` block with the updated version
   - Keep drafts concise and appropriate for the platform (tweet-length for Twitter, longer for LinkedIn, etc.)
   - Don't include `<draft>` markers unless the user is asking for a draft or refinement

Merge voice profile: if `manualEdits` exists, deep-merge with `extractedProfile` (manualEdits values override extracted values). Use a simple recursive merge — this mirrors the Phase 2 display-time merge pattern.
  </action>

**Create `src/routes/(app)/queue/[id]/chat/+server.ts`** — the streaming chat endpoint:

1. **Auth check**: Use `locals.auth()` — return 401 if no session (same pattern as voice/+server.ts).

2. **Parse request body**: The DefaultChatTransport sends JSON `{ id, messages, trigger, messageId, ...body }`. Extract `messages` (UIMessage[]) and `personaId` (from spread body).

3. **Load context from DB** (per-request DB, same pattern):
   - Get the post via `queueService.getById(userId, params.id)` — 404 if not found
   - Determine persona: use `personaId` from request body if present, else `post.personaId`
   - Load voice profile via `voiceService.getActiveVersion(personaId)` if personaId exists
   - Load persona name via `personaService.getById(userId, personaId)` if personaId exists

4. **Persist new user messages**: Get existing message IDs via `chatService.getMessageIds(params.id)`. For each message in the request that isn't in the DB and has role 'user', save it via `chatService.saveMessage()`. This persists user messages BEFORE streaming (so they're saved even if streaming fails).

5. **Status transition**: If the post status is 'new', update to 'in_progress' via `queueService.updateStatus()`.

6. **Build system prompt**: Call `buildChatSystemPrompt()` with loaded context.

7. **Convert messages**: Use `convertToModelMessages(messages)` from 'ai' package to convert UIMessage[] to model-compatible format.

8. **Stream**: Create Anthropic provider with API key from `platform?.env?.ANTHROPIC_API_KEY ?? env.ANTHROPIC_API_KEY`. Call `streamText()` with model `anthropic('claude-sonnet-4-20250514')`, system prompt, converted messages, maxTokens 2048.

9. **Return stream**: `result.toUIMessageStreamResponse({ onFinish })` where onFinish:
   - Receives `{ responseMessage }` — this is the complete AI UIMessage
   - Persist the responseMessage via `chatService.saveMessage(params.id, { id: responseMessage.id, role: responseMessage.role, parts: responseMessage.parts })`
   - Check if any text part contains `<draft>` — if so and post status is 'in_progress', update status to 'draft_ready'
   - If personaId from body differs from post.personaId, update the post's persona via `queueService.updatePersona()`

**Update `src/routes/(app)/queue/[id]/+page.server.ts`** — enhance the page load:

Keep the existing `load` function structure and `actions` (updateContent). Add to the load function:

1. Load chat messages: `chatService.getMessages(post.id)` — returns ordered messages
2. Load voice profile: if post has a personaId, load active voice version via `voiceService.getActiveVersion(post.personaId)`
3. Load all non-archived personas for the user: `personaService.list(userId)` — needed for persona switcher dropdown
4. Load draft edits: `draftService.getEditsForPost(post.id)` — needed to show edited versions in DraftBlocks

Format chat messages as UIMessage-compatible objects for the Chat class: `{ id, role, parts, createdAt }`. The `parts` field is already stored as JSONB in the correct format.

Return: `{ post, persona, chatMessages, voiceProfile, personas, draftEdits }`

Also add `createVoiceService`, `createChatService`, and `createDraftService` to the getServices helper.

Generate an initial greeting message if chatMessages is empty: create a template assistant message with text "Ready to help you draft a reply to {postAuthor}'s post as {personaName}. What angle do you want to take?" — save it to the DB and include it in the returned messages. Use generic fallback text if no author/persona.
