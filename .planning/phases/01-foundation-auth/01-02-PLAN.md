  <action>
    1. **Create src/routes/(auth)/+layout.svelte** — Minimal auth layout:
       - Centered content container (Tailwind: flex, items-center, justify-center, min-h-screen, bg-gray-50 or similar neutral)
       - Renders `{@render children()}` (Svelte 5 syntax, NOT `<slot />`)
       - No navigation — auth pages are standalone

    2. **Create src/routes/(auth)/register/+page.server.ts** — Registration form action:
       - Import `signIn` from `$root/auth` (or wherever auth.ts exports it — may need to use `../../auth` or a SvelteKit alias).
       - Import `registerSchema` from `$lib/schemas/auth`.
       - Import `hashPassword` from `$lib/server/auth/password`.
       - Import `createDb` from `$lib/server/db` and `users` from `$lib/server/db/schema`.
       - `load` function: Check if already authenticated via `event.locals.auth()`. If session exists, `redirect(303, "/")`.
       - `actions.default`: Parse formData with `registerSchema.safeParse()`. On validation failure, return `fail(400, { errors, email })` preserving the email value.
       - Check for duplicate email: query DB for existing user with that email. If found, return `fail(400, { errors: { email: ["Email already registered"] }, email })`.
       - Hash password with `hashPassword(password)`.
       - Insert user into DB: `db.insert(users).values({ email, passwordHash, name: email.split("@")[0] })`.
       - After user creation, call `await signIn("credentials", { email, password, redirectTo: "/" })` passing the event. This signs them in via Auth.js.
       - **IMPORTANT:** The DB instance must be created per-request on Workers. Use `createDb()` with the DATABASE_URL from `event.platform?.env?.DATABASE_URL ?? process.env.DATABASE_URL ?? ""`.

    3. **Create src/routes/(auth)/register/+page.svelte** — Registration form UI:
       - Use Svelte 5 runes: `let { form } = $props()` for form action return data.
       - Form with `method="POST"` and `use:enhance` (import from `$app/forms`).
       - Fields: email (type="email", required), password (type="password", required, minlength=8), confirmPassword (type="password", required).
       - Inline error display below each field using `form?.errors?.fieldName`.
       - General error display for `form?.error` (non-field errors).
       - Submit button: "Create account".
       - Link to login page: "Already have an account? Sign in".
       - Style with Tailwind: card-like container (white bg, rounded, shadow, padding), proper spacing, accessible labels.
       - Preserve email value on error: `value={form?.email ?? ""}`.

    4. **Create src/routes/(auth)/login/+page.server.ts** — Login form action:
       - `load` function: Check if already authenticated. If session exists, `redirect(303, "/")`.
       - `actions.default`: Parse formData with `signInSchema.safeParse()`. On validation failure, return `fail(400, { errors, email })`.
       - Call `await signIn("credentials", { email, password, redirectTo: "/" })` passing the event.
       - **Handle Auth.js errors:** Auth.js signIn with Credentials throws `AuthError` when authorize returns null. Catch this and return `fail(400, { error: "Invalid email or password", email })`.
       - Import `AuthError` from `@auth/sveltekit` (or `@auth/core/errors`) for proper error type checking.

    5. **Create src/routes/(auth)/login/+page.svelte** — Login form UI:
       - Same pattern as register page but simpler: email + password fields only.
       - `let { form } = $props()`.
       - Inline field errors + general error message.
       - Submit button: "Sign in".
       - Link to register page: "Don't have an account? Create one".
       - Style consistently with register page.

    **Svelte 5 REMINDERS:**
    - Use `$props()` not `export let`
    - Use `{@render children()}` not `<slot />`
    - Use `onclick` not `on:click`
    - Use `$state()` for reactive local state if needed

<task type="auto">
  <name>Task 2: Create protected app shell with session loading, auth guard, and logout</name>
  <files>
    src/routes/+layout.svelte

  <action>
    1. **Create src/routes/+layout.server.ts** — Root session loader:
       - Load session via `event.locals.auth()` and return it.
       - This makes session available to ALL pages (both auth and app groups).
       - ```typescript
         export const load = async (event) => {
           const session = await event.locals.auth()
           return { session }
         }
         ```

    2. **Create src/routes/+layout.svelte** — Root layout:
       - Import `src/app.css` for Tailwind styles.
       - Render `{@render children()}`.
       - Minimal wrapper — no navigation here (auth pages don't need nav, app pages get their own).

    3. **Create src/routes/(app)/+layout.server.ts** — Auth guard:
       - Load session via `event.locals.auth()`.
       - If no session or no session.user, `redirect(303, "/login")`.
       - Return `{ session }` for child routes.
       - This protects ALL routes under `(app)/` — any unauthenticated access redirects to login.

    4. **Create src/routes/(app)/+layout.svelte** — App shell layout:
       - `let { data, children } = $props()`.
       - Simple top navigation bar with: app name "ez-social" (left), user email (right), logout form (right).
       - Logout: Use a `<form method="POST" action="/auth/signout">` — Auth.js handles the `/auth/signout` route automatically. Include a hidden CSRF token if needed, or use the Auth.js signOut approach. **Simplest approach:** `<form method="POST" action="/auth/signout"><button type="submit">Sign out</button></form>`. Auth.js's handle hook processes this route.
       - Below nav: render `{@render children()}` in a main content area.
       - Style: clean, minimal nav (white bg, border-bottom, flex justify-between), content area with max-width and padding.

    5. **Create src/routes/(app)/+page.svelte** — Protected home page (placeholder):
       - `let { data } = $props()`.
       - Display: "Welcome to ez-social" heading.
       - Show user's email from `data.session.user.email`.
       - Brief placeholder text: "Your social media response assistant. Features coming soon."
       - This is the page users land on after login/register.
       - Style: centered content, clean typography.

    **Auth.js logout notes:**
    - Auth.js automatically handles POST to `/auth/signout` (or the configured signOut path).
    - After signOut, Auth.js clears the session cookie and redirects.
    - The redirect after signOut goes to the page configured in Auth.js or defaults to `/`.
    - Since `/` is protected (inside `(app)/`), an unauthenticated user hitting `/` will be redirected to `/login` by the auth guard. So the chain is: signOut → `/` → redirect to `/login`. This is correct behavior.

    **Session access pattern:**
    - Auth.js with JWT strategy: `event.locals.auth()` returns `{ user: { name, email, image, id } }` or `null`.
    - The `id` is available because we add it in the JWT callback (configured in Plan 01 Task 3).
  </action>

Refresh the page — session persists (no re-login required, AUTH-03). Click "Sign out" — should log out and redirect to `/login`. Try to navigate back to `/` without logging in — redirected to `/login` again.
  </verify>
  <done>
    Root layout loads session. App route group has auth guard redirecting unauthenticated users. Protected home page shows user info. Logout works via Auth.js signOut route. Session persists across browser refresh (JWT in HttpOnly cookie).
