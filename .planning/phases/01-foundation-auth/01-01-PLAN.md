  <action>
    1. **Create src/lib/server/db/schema.ts** — RECONCILE architecture.md schema with Auth.js requirements:
       - `users` table: `id` (uuid, defaultRandom, PK), `name` (varchar 255, nullable), `email` (varchar 255, unique, notNull), `emailVerified` (timestamp w/tz, nullable), `image` (text, nullable), `passwordHash` (varchar 255, nullable — custom field for credentials auth), `createdAt` (timestamp w/tz, defaultNow), `updatedAt` (timestamp w/tz, defaultNow)
       - `accounts` table: Auth.js standard — `userId` (uuid, FK → users, cascade), `type` (varchar), `provider` (varchar), `providerAccountId` (varchar), `refresh_token`, `access_token`, `expires_at` (integer), `token_type`, `scope`, `id_token`, `session_state`. Composite PK on (provider, providerAccountId).
       - `sessions` table: Auth.js standard — `sessionToken` (varchar, PK), `userId` (uuid, FK → users, cascade), `expires` (timestamp). Note: even though we use JWT strategy, include this table for future flexibility.
       - `verificationTokens` table: Auth.js standard — `identifier` (varchar), `token` (varchar), `expires` (timestamp). Composite PK on (identifier, token).
       - Relations: users → many sessions, users → many accounts.
       - **DO NOT** include personas, voiceProfiles, postQueue, drafts, or auditLog tables — those belong to future phases. Only auth-related tables.
       - Use `pgTable` from `drizzle-orm/pg-core`. Follow exact Auth.js Drizzle adapter column naming conventions so the adapter works without custom mapping.

    2. **Create src/lib/server/db/index.ts** — DB connection factory:
       - Export `createDb(databaseUrl: string)` function that creates a Neon HTTP client and returns a Drizzle instance with schema.
       - Use `drizzle` from `drizzle-orm/neon-http` and `neon` from `@neondatabase/serverless`.
       - This is the ONLY way to get a DB instance — no module-scope singletons (Workers are stateless per-request).

    3. **Create src/lib/server/auth/password.ts** — Workers-compatible password hashing:
       - Use Web Crypto API `PBKDF2` (built into Workers runtime, zero dependencies).
       - `hashPassword(password: string): Promise<string>` — Generate 16-byte random salt, derive key with PBKDF2 (SHA-256, 100,000 iterations, 32-byte output), return as `"100000:base64salt:base64hash"` string.
       - `verifyPassword(stored: string, password: string): Promise<boolean>` — Parse stored string, re-derive hash with same params, constant-time comparison.
       - Use `crypto.getRandomValues` for salt, `crypto.subtle.importKey` + `crypto.subtle.deriveBits` for PBKDF2.
       - Implement proper constant-time comparison to prevent timing attacks.
       - Follow the exact pattern from 01-RESEARCH.md Pattern 4.

    4. **Create src/lib/schemas/auth.ts** — Zod validation schemas:
       - `signInSchema`: email (string, required, valid email), password (string, required)
       - `registerSchema`: email (string, required, valid email), password (string, min 8, max 72), confirmPassword (string). Refine: passwords must match.
       - Export both schemas and their inferred types.
  </action>

  <action>
    1. **Create src/auth.ts** — Auth.js SvelteKit configuration:
       - Import `SvelteKitAuth` from `@auth/sveltekit`.
       - Import `Credentials` from `@auth/sveltekit/providers/credentials`.
       - Import `DrizzleAdapter` from `@auth/drizzle-adapter`.
       - Use the **callback pattern** for Workers: `SvelteKitAuth(async (event) => { ... })` so the DB can be initialized per-request from `event.platform?.env?.DATABASE_URL ?? process.env.DATABASE_URL`.
       - Inside the callback, create DB via `createDb(databaseUrl)` and return the full config object.
       - Adapter: `DrizzleAdapter(db, { usersTable: users, accountsTable: accounts, sessionsTable: sessions, verificationTokensTable: verificationTokens })` — pass custom table references to map to our schema.
       - Provider: `Credentials` with email and password fields. The `authorize` callback: parse credentials with `signInSchema`, query user by email from DB, return null if not found, verify password with `verifyPassword`, return null if invalid, return `{ id, email, name }` if valid.
       - `session: { strategy: "jwt" }` — JWT strategy, no DB session queries per request.
       - `pages: { signIn: "/login" }` — Custom login page.
       - `trustHost: true` — Required for Cloudflare Workers.
       - `callbacks.jwt`: Add user.id to the JWT token on initial sign-in (when `user` is present).
       - `callbacks.session`: Add user.id from token to the session object so it's available client-side.
       - Export `{ handle, signIn, signOut }` from `SvelteKitAuth(...)`.

    2. **Update src/hooks.server.ts**:
       - Re-export `handle` from `./auth`: `export { handle } from "./auth"`.
       - Keep it simple — Auth.js handle does everything (session management, CSRF, cookie handling).

    3. **Update src/app.d.ts** (may already exist from Task 1):
       - Ensure `App.Locals` does NOT manually declare `user`/`session` — Auth.js manages this via `event.locals.auth()` which returns `Session | null`.
       - Ensure `App.Platform` has `env?: { DATABASE_URL: string; AUTH_SECRET: string }`.
       - Import and reference the Auth.js session type if needed.

    **IMPORTANT:**
    - Do NOT use `@auth/sveltekit/client` components — use server-side form actions for sign-in/sign-out (per research recommendation, avoids Svelte 5 compatibility issues).
    - The `authorize` callback ONLY handles login verification. Registration is a separate concern (Plan 02).
    - Auth.js Credentials provider does NOT auto-create users in the DB. The adapter handles OAuth users, but Credentials users must be created manually via registration.
